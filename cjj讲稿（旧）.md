## 目标一：Zigbee 协调器任务切换（.h文件最后）代码event_

### 讲稿

协调器主要有四个主要任务存在，分别是 `系统初始化任务`、`SPI 数据接收任务`、`Zigbee 数据处理任务` 和 `SPI 队列发送任务`

可以看到在初始化结束后，紧接的的便是这 3 个核心任务，任务按照优先级和触发条件进行相互切换。

优先级是最高的是`SPI 数据接收任务` 。当系统检测到 SPI 数据到达，就会优先进入本任务，对接收的数据按照协议进行解码。原因是上位机或网关下发过来的命令，需要先被及时接收，否则后面的处理和执行都无从谈起。

解码完成后，如果这是一条需要下发给 Zigbee 网络节点的命令，协调器就会把它下发到对应的 Zigbee 节点。节点执行后，会返回相应的反馈数据。

接下来，系统会进入 `Zigbee 数据处理任务`。此任务主要就是将对节点上传的数据或者节点返回的响应结果进行解析和整理。如有必要，还会把这些数据进一步转换成适合 SPI 通信协议的格式，为后续上送做准备。

最后便是 `SPI 队列发送任务`。它主要负责把已经处理好的数据，从发送队列中按顺序发回到上级设备。如果当前队列为空，没有有效数据发送，便会发送一个心跳包，告诉上位机，协调器仍然处于正常在线状态。

### 总结

这一整段代码，使协调器通过 3 个任务的切换，实现数据接收、处理和发送的解耦，提高了系统整体实时性与稳定性。

## 目标二：Zigbee 协调器时间片调度算法

### 讲稿

在 Zigbee 网络中，如果多节点在同一时间集中上报数据，很容易出现信道冲突、数据堆积，严重时还会影响命令下发和节点反馈的实时性。为了解决这个问题，此文件c代码便是我们引入的时间片调度算法。

它的逻辑是，当某个 Zigbee 节点初次入网时，它并不是立即无序地发送数据，而是先向协调器申请相关的调度参数。协调器在判断该节点是初次入网后，就会执行此算法，为这个节点分配对应的发送时隙。

这样一来，不同节点的数据发送时间就被错开。原本可能会集中在某一时刻发送的数据，被分散到了不同的时间段，从而减少了多个节点同时占用信道的情况。

这种设计带来的效果，就像我们前面所说的：

第一，它能够有效减少网络拥塞，避免因为节点同时发送数据而导致的通信冲突。

第二，它能够提高系统整体的实时性和稳定性，使协调器收到的数据更加均匀，也更方便后续的数据处理和转发。

### 总结

协调器通过我们的时间片调度算法，让节点按分配的时间片错峰发送数据，避免网络拥塞，提高数据传输时的实时性和稳定性。

## 目标三：和校验函数边敲边讲

### 代码

```c
char Checksum(char *data)
{
  int temp = 0;
  for (int i = 1; i < 7; i++)
  {
    temp = temp + data[i];
  }
  return temp & 0xFF;
}
```

### 讲稿

为了确保数据在传输前后的正确性++

这里我编写的是一个 `Checksum` 函数，它的作用是对一帧数据中的有效字段进行和校验，用于判断数据在传输过程中是否发生了错误。

接下来，我定义了一个整型变量 `temp`，并把它初始化为 0。这个变量是作为累加器，把需要参与校验的每个字节依次相加。

然后进入 `for` 循环。这里循环是从 1 到 6，也就是从帧的有效字段开始。

因为在通信协议里，`data[0]` 往往是帧头，它主要是标识一帧数据开始，一般不参与校验。同样，校验位本身和帧尾也不参与本次求和，所以只对中间的有效数据字段进行累加。

在循环内部，就是把每一个有效字节累加到 `temp` 中。

最后，函数返回 `temp & 0xFF`，保留累加结果的低 8 位，作为最终的校验和。

发送端在发送数据时，会先按照这个规则计算出校验和，并把它放入数据帧中；接收端收到数据后，再按相同的方法重新计算。如果两个结果一致，就说明这一帧数据基本是正确的；反之，则说明数据在传输过程中可能出现了错误。
